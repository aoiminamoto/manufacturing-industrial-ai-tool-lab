# Security and Intellectual-Property Boundary

## Public in this project

- Clean-room code written for this portfolio
- General architecture patterns
- Synthetic Japanese-English examples
- Generic PLC, robot, HMI, and structured-file profiles
- Automated tests and public architecture decisions
- A retrospective timeline with sanitized descriptions

## Kept private

- Employer, customer, supplier, plant, program, and equipment identities
- Logos, branded interfaces, and internal screenshots
- Real terminology standards, glossaries, source files, and translated outputs
- Credentials, API configuration, endpoints, network details, and server paths
- Internal usage logs, databases, job artifacts, and operational records
- Proprietary source code, prompts, requirements, and deployment configuration
- Names of reviewers or managers without their permission

## Publishing rule

A concept may be made public only when it can be explained and demonstrated with newly written code and synthetic data. When the evidence depends on a private artifact, the public repository records only the evidence category and status; the artifact remains in a controlled private evidence file.

## Claim rule

Use measured, dated, and independently supportable claims. If a metric or confirmation is unavailable, label it `Not yet measured` or `Not yet attached` rather than estimating it.
